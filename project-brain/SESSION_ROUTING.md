# ReplyOS — WhatsApp Session Routing & Synchronization

This document specifies the companion device lifecycle, socket initialization, database-backed credential storage, and session termination pathways.

---

## 1. Session Lifecycle States

Every WhatsApp socket session is monitored dynamically in the platform:

```
                  +--------------------------------+
                  |         disconnected           |
                  +--------------------------------+
                    │                            ▲
       User clicks  │                            │ Companion
       Connect / QR │                            │ Logged Out / Severed
                    ▼                            │
                  +--------------------------------+
                  |          connecting            |
                  +--------------------------------+
                    │                            ▲
         QR Scan    │                            │ Network Drops
        Succeeds    │                            │ (Starts auto-retry)
                    ▼                            │
                  +--------------------------------+
                  |           connected            |
                  +--------------------------------+
```

---

## 2. DB-Backed Credentials Synchronization

To prevent data loss when container instances are rebuilt or moved, credentials are saved inside PostgreSQL:
* **Storage Model**: Raw authentication credential states (cookies, tokens, keys) generated by the Baileys client-side engine are parsed to JSON and saved inside the `whatsapp_sessions.session_auth_data` column.
* **Encryption standard**: Before committing, credential dictionaries are encrypted in Python using **AES-256-GCM** using the environment-supplied secret key `SESSION_ENCRYPTION_KEY`.
* **Sync Frequency**: On every credentials modification callback triggered by Baileys, the Node companion relays the updated authorization structure to the FastAPI backend, which updates the database.

---

## 3. QR Generation and Hydration

* **Trigger**: When the client clicks "Connect" on `/dashboard`, the backend calls `POST /sessions` to create a record in the database.
* **Webhooks integration**: The backend instructs the Node engine to initialize a socket connection for the session ID. 
* **QR Extraction**: Baileys triggers a `connection.update` event carrying raw QR strings. The Node engine relays QR codes back to the FastAPI `/sessions/webhook` receiver, which commits the strings to `whatsapp_sessions.qr_code`.
* **UI Hydration**: The React frontend polls `/api/v1/sessions/{id}` or listens on the WebSocket channel to render the dynamic QR canvas.

---

## 4. Session Revocation (Disconnect Protocols)

To safely delete connections without leaving orphan records, a strict multi-tier disconnect protocol is enforced:
1. **User Action**: Clicking "Disconnect Session" calls `POST /api/v1/sessions/{id}/revoke-sessions` (Admin) or `DELETE /api/v1/sessions/{id}` (Customer).
2. **Engine severing**: The backend makes an API call to `/sessions/{id}` on the Node engine, prompting Baileys to call `sock.logout()` and sever the companion socket cleanly.
3. **DB updates**: Updates status in `whatsapp_sessions` to `"disconnected"`, clears the encrypted `session_auth_data` column, and wipes the cached QR strings.
