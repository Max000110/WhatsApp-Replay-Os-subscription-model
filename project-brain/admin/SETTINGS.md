# Master Admin Settings Specification

This document details the configuration options and security boundaries accessible within the Super Admin Settings control panel.

---

## 1. Administrative Account Rotation

Super Admin profile identity is bound to `admin@replyos.com` with `super_admin` scopes. 

### Password Rotation Policies
* **Complexity**: Passwords must be at least 8 characters long, containing alphanumeric characters.
* **Must Change Flag**: The database seeds the initial account with `must_change_password = True`. Login attempts block access to all downstream tenant registries with `403 Forbidden` until a new secure passcode is locked.
* **JWT Token Lifetime**: Standard session token expiration is set to 2 hours. Rotating the passcode blacklists the previous token in Redis immediately and issues a newly signed claims token containing the updated password validation hash.

---

## 2. Multi-Tenant MFA Hardening

TOTP Two-Factor Authentication (2FA) restricts administrative access to the control plane, utilizing a pure-python drift-aware verification mechanism.

### Key Setup & Verification
1. **Secret Generation**: Generates a 20-byte cryptographically secure random key, encoding it as Base32.
2. **Dynamic QR/URI Mapping**: Creates an RFC 6238 compliant OTPAuth URI:
   `otpauth://totp/ReplyOS-Admin:[email]?secret=[secret]&issuer=ReplyOS`
3. **Drift Compensation**: The core validation allows a clock skew offset of +/- 30 seconds (1 time-step drift) to support unsynchronized devices.
4. **Recovery Keys**: Enforcing 2FA automatically generates eight distinct 8-digit hexadecimal recovery codes. Entering a recovery code consumes it from the database table and successfully signs the JWT with `totp_verified = True`.

---

## 3. Telemetry & Audit Configurations

Administrators can toggle auditing parameters to balance performance and compliance bounds.

* **Audit Retention Mode**: Configure database archiving or progressive cleanups.
* **Notification Routing**: Toggle SMTP email dispatch triggers on automatic grace suspensions or rate-limiting bans.
* **Impersonation Auditing**: Any tenant troubleshooting session generated by the impersonator controller writes a permanent audit record containing the admin operator, target tenant, session timestamp, and client IP constraints.
