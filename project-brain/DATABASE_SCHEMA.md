# ReplyOS — Database Schema Specification

This document details the tables, relations, indexes, and unique constraints implemented in the PostgreSQL database.

---

## 1. Schema Diagram (Entities & Relationships)

```
 [tenants] 1 ──── 0..* [users]
    │ 1
    ├──────────── 0..* [whatsapp_sessions] 1 ── 0..* [conversations] 1 ── 0..* [messages]
    │ 1                                                                       │ 1..*
    ├──────────── 0..* [chatbots] ────────────────────────────────────────────┘
    │ 1
    ├──────────── 1 [subscriptions]
    │ 1
    └──────────── 0..* [audit_logs]
```

---

## 2. Table Definitions (SQL DDL Mappings)

### 2.1 `tenants`
Stores platforms tenants and domains.
* `id` (UUID, PK): Auto-generated.
* `name` (VARCHAR): Tenant business name.
* `subdomain` (VARCHAR): Unique subdomain indicator.
* `status` (VARCHAR): Current lifecycle (`active`, `suspended`, `terminated`).
* `data_retention_policy` (VARCHAR): Data policy (`archive`, `delete`).
* `termination_grace_period_ends` (TIMESTAMP): 24h grace expiration marker.

### 2.2 `users`
System user accounts.
* `id` (UUID, PK)
* `tenant_id` (UUID, FK -> `tenants.id` on delete CASCADE)
* `email` (VARCHAR, UNIQUE): Account email.
* `password_hash` (VARCHAR): Bcrypt hash.
* `role` (VARCHAR): Access level (`admin`, `owner`, `member`).
* `is_active` (BOOLEAN): Active status indicator.

### 2.3 `subscriptions`
Tenant billing and plan states.
* `id` (UUID, PK)
* `tenant_id` (UUID, FK -> `tenants.id` on delete CASCADE)
* `plan_tier` (VARCHAR): Plan key (`free`, `starter`, `pro`, `agency`).
* `status` (VARCHAR): Subscription billing status (`active`, `suspended`).
* `max_bots` (INTEGER): Hard quota cap for active bots.
* `max_messages` (INTEGER): Hard quota cap for monthly broadcasts.
* `current_period_end` (TIMESTAMP): Expiry date marker.

### 2.4 `whatsapp_sessions`
Baileys multi-device socket credentials and session records.
* `id` (UUID, PK)
* `tenant_id` (UUID, FK -> `tenants.id` on delete CASCADE)
* `session_name` (VARCHAR): Session indicator.
* `phone_number` (VARCHAR): Linked WhatsApp number JID.
* `status` (VARCHAR): Session status (`connected`, `disconnected`).
* `session_auth_data` (JSON): AES-256-GCM encrypted Baileys tokens.

### 2.5 `conversations`
Messaging threads linked to customers.
* `id` (UUID, PK)
* `tenant_id` (UUID, FK -> `tenants.id` on delete CASCADE)
* `session_id` (UUID, FK -> `whatsapp_sessions.id` on delete CASCADE)
* `customer_phone` (VARCHAR): Normalized phone JID.
* `bot_paused_until` (TIMESTAMP): human override takeover timer.

> [!NOTE]
> Enforces JID uniqueness using constraint: `uq_tenant_customer_phone UNIQUE (tenant_id, customer_phone)`.

### 2.6 `messages`
Individual message records.
* `id` (UUID, PK)
* `conversation_id` (UUID, FK -> `conversations.id` on delete CASCADE)
* `whatsapp_message_id` (VARCHAR, UNIQUE): Unique callback key.
* `sender` (VARCHAR): Sender phone number or `"bot"`.
* `text` (TEXT): Message body.
* `origin` (VARCHAR): Message direction (`inbound`, `outbound`).
* `ack_state` (VARCHAR): Delivery receipts (`queued`, `sent`, `delivered`, `read`).

### 2.7 `kb_document_chunks`
Vector chunks for RAG searching.
* `id` (UUID, PK)
* `document_id` (UUID, FK -> `kb_documents.id` on delete CASCADE)
* `content` (TEXT): Text slice.
* `embedding` (Vector(384)): Vector embeddings generated by Ollama for search.

### 2.8 `audit_logs`
Super admin actions audit log.
* `id` (UUID, PK)
* `admin_user_id` (UUID, FK -> `users.id`, NULLABLE): Action perpetrator.
* `action_type` (VARCHAR): Logged action key (`SUSPEND_TENANT`, `CHANGE_PLAN`).
* `affected_resources` (VARCHAR): Resource key (`subscription`, `users`).
* `new_state` (JSON): Details of changes.
* `timestamp` (TIMESTAMP): Auto-generated.
