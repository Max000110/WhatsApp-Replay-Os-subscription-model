# JID Root Cause Analysis — ReplyOS
**Date**: 2026-05-29 | **Prepared by**: Principal WhatsApp Infrastructure Engineer

---

## 1. Executive Summary

This audit investigates JID format handling across the ReplyOS messaging pipeline. Specifically, we trace why the 15-digit ID `185654373789739@s.whatsapp.net` was generated, find the responsible codebase locations, and clarify the separation of core WhatsApp message key identifiers.

---

## 2. JID Generation: Root Cause Analysis

### 2.1 Where and Why Generated
The JID `185654373789739` was generated as a result of an inbound customer message received by the Baileys session engine.
* **Original Message Origin**: A WhatsApp user with a **Line Identity (LID)** JID: `185654373789739@lid`.
* **WhatsApp LID Definition**: LIDs are used by WhatsApp for community-based messaging to prevent exposing physical phone numbers to other community members.
* **Why `@s.whatsapp.net` got appended**: The Node.js Baileys engine stripped the `@lid` domain when conveying the phone identifier to the backend webhook via the `from` field (which was sent as just `185654373789739`). The FastAPI backend's `normalize_jid` function saw a domain-less number and defaulted to appending `@s.whatsapp.net`.

### 2.2 Forensic Code Trace

1. **Baileys Engine Upsert Handshake**:
   In `whatsapp-engine/src/baileys-manager.ts` at line 170-172:
   ```typescript
   const remoteJid = msg.key.remoteJid; // "185654373789739@lid"
   const from = remoteJid?.split("@")[0].split(":")[0] || ""; // "185654373789739"
   ```
   *Note: This strips the `@lid` domain suffix.*

2. **Webhook Dispatch**:
   The engine forwarded this payload via POST to `/api/v1/sessions/webhook`:
   ```json
   {
     "from": "185654373789739",
     "rawRemoteJid": "185654373789739@lid"
   }
   ```

3. **FastAPI Webhook Resolver**:
   In `backend/app/routers/sessions.py` (prior to patch):
   ```python
   raw_from = data_dict.get("from", "")
   customer_phone = normalize_jid(raw_from) # normalize_jid("185654373789739")
   ```

4. **JID Normalizer Execution**:
   In `backend/app/core/jid.py`:
   ```python
   # Since raw input lacks '@', domain falls back to default:
   domain = "s.whatsapp.net"
   ...
   return f"{user}@{domain}" # "185654373789739@s.whatsapp.net"
   ```

5. **Resulting Database Impact**:
   A conversation was created in the DB with JID `185654373789739@s.whatsapp.net` instead of `185654373789739@lid`. This caused replies to stay in `sent` status forever because the `@s.whatsapp.net` destination JID was invalid for that user.

## 3. Separation of WhatsApp Schema Identifiers

To prevent data corruption, the system enforces strict boundaries between key identifiers. They are never confused in database columns or API routing payloads:

| Identifier | Class Type | System Purpose | Format Example |
|:---|:---|:---|:---|
| **`remoteJid`** | `str` | Canonical WhatsApp destination address. | `917021886525@s.whatsapp.net` or `185654373789739@lid` |
| **`participant`** | `str` | Group message author (only present in group events). | `919137730283@s.whatsapp.net` |
| **`message.key.id`** | `str` | Unique WhatsApp network message ID (generated by WhatsApp server). | `3EB015B5AEFB2E6DFE3989` |
| **`conversation_id`** | `UUID` | Local PostgreSQL primary key identifying a user thread. | `0ee65fe3-ab01-45c4-8c9f-0cb549c02570` |
| **`thread_id`** | `UUID` | Synonym for conversation ID. | `0ee65fe3-ab01-45c4-8c9f-0cb549c02570` |

---

## 4. Normalization Rules & Resolution

The platform enforces the following canonical normalization guidelines:
1. **Indian Numbers**: Strict 10-digit validation with auto-prepend: `7021886525` → `917021886525@s.whatsapp.net`.
2. **International Numbers**: General fallback for length 7-20 digits.
3. **Domain Conservation**: `@lid`, `@s.whatsapp.net`, and `@g.us` domains are fully preserved.
4. **Stripping Colons**: Device indices (e.g. `:12`) are stripped before normalizing the JID: `917021886525:12@s.whatsapp.net` → `917021886525@s.whatsapp.net`.

### Correction Applied
The FastAPI webhook receiver was patched to read `rawRemoteJid` (with domain intact) rather than `from` (with domain stripped):
```python
raw_from = data_dict.get("rawRemoteJid") or data_dict.get("from", "")
customer_phone = normalize_jid(raw_from)
```
This guarantees that `@lid` formats are correctly normalized and stored in the database, allowing successful live override dispatches.
